# Android Patch Workflow

## Purpose
This document defines how local modifications to the upstream NekoBox fork are created, named, ordered, applied, and re-applied.
It is the authoritative reference for the patch workflow in `android/patches/`.

## Patch Philosophy
Patches are the mechanism for separating upstream code from local project modifications.
Every local change to fork content must be expressible as a discrete, named patch.
This keeps the upstream snapshot clean and makes re-application after an upstream sync tractable.

## Patch Storage
All patches live in `android/patches/`.
Patches are standard unified diff files generated by `git diff` or `git format-patch`.

## Patch Naming Convention

```
<sequence>-<scope>-<short-description>.patch
```

| Component          | Rule                                                       |
|--------------------|------------------------------------------------------------|
| `<sequence>`       | Three-digit zero-padded integer starting at `001`          |
| `<scope>`          | Short category: `routing`, `transport`, `ui`, `build`, `config`, `core` |
| `<short-description>` | Lowercase, hyphen-separated, max ~40 characters        |

Examples:
- `001-build-add-naive-proxy-dependency.patch`
- `002-routing-per-app-policy-hooks.patch`
- `003-config-custom-server-endpoint.patch`

The sequence number defines application order. Gaps are allowed (for future insertion) but application must follow ascending numeric order.

## Patch Creation Procedure

1. **Start from a clean post-intake state.**
   Ensure `android/fork/` matches the last committed upstream snapshot with all prior patches applied.

2. **Make the local modification** directly in the relevant files under `android/fork/`.

3. **Generate the patch.**
   ```powershell
   # From the repository root, after staging the change
   git diff android/fork/ > android/patches/<sequence>-<scope>-<short-description>.patch
   ```
   Or, if using `git format-patch` for a committed change:
   ```powershell
   git format-patch -1 HEAD --output-directory android/patches/ --start-number <sequence>
   # Rename the output file to match the naming convention
   ```

4. **Add a header comment** to the top of the patch file:
   ```
   # Patch: <sequence>-<scope>-<short-description>
   # Author: <name or session identifier>
   # Date: <YYYY-MM-DD>
   # Checkpoint: <CP-NNN>
   # Description: <one-line summary of what this patch does and why>
   ```

5. **Commit the patch file.**
   ```powershell
   git add android/patches/<patch-file>
   git commit -m "patch(android): add <sequence>-<scope>-<short-description>"
   ```

## Patch Application Procedure

To apply all patches in order against a clean upstream snapshot:

```powershell
# From the repository root
$patches = Get-ChildItem android/patches/*.patch | Sort-Object Name
foreach ($patch in $patches) {
    git apply --directory=android/fork $patch.FullName
    if ($LASTEXITCODE -ne 0) {
        Write-Host "FAILED to apply: $($patch.Name)" -ForegroundColor Red
        Write-Host "Resolve the conflict manually, then continue with the next patch."
        break
    }
    Write-Host "Applied: $($patch.Name)" -ForegroundColor Green
}
```

After all patches are applied, stage and commit:
```powershell
git add android/fork/
git commit -m "patch(android): apply all local patches on upstream snapshot"
```

## Patch Re-Application After Upstream Sync

When the upstream snapshot is updated (see `docs/android/ANDROID_FORK_INTAKE.md`, Upstream Sync Procedure):

1. Begin from the freshly ingested upstream snapshot (no patches applied).
2. Run the patch application procedure above.
3. If a patch fails to apply cleanly:
   - Inspect the conflict.
   - Manually resolve the conflict in `android/fork/`.
   - Regenerate the patch from the resolved state.
   - Replace the old patch file in `android/patches/` with the updated version.
   - Document the conflict and resolution in the patch header comment.
4. After all patches are resolved and applied, commit the updated fork state and updated patch files together.

## Patch Index

Maintain `android/patches/PATCH_INDEX.md` (created when the first patch is added) with a table:

| Sequence | Scope     | Description                    | Checkpoint | Status  |
|----------|-----------|--------------------------------|------------|---------|
| 001      | build     | Add NaiveProxy dependency      | CP-NNN     | pending |

Status values: `pending`, `applied`, `conflict`, `retired`.

This file is not yet created because no patches exist. It will be created in the checkpoint that produces the first patch.

## Verification
The fork intake verification script (`android/scripts/verify-fork-intake.ps1`) checks patch baseline consistency:
- `android/patches/README.md` exists
- patch files (if any) follow the naming convention
- no patch files exist outside `android/patches/`

## Explicit Non-Goals of This Document
- This document does not create any patches.
- This document does not modify fork content.
- This document does not implement Android application behavior, per-app routing, or transport logic.
