# CP-012

## CHECKPOINT_ID
CP-012

## TITLE
Attempt the smallest tool-local repair for the libcore zero-byte generated `go.mod` blocker

## CURRENT_STATE
CP-011 is complete.
The libcore gomobile blocker is localized to zero-byte generated `go.mod` files under `android/fork/libcore/.build/src-android-*`.
The first exact failing generated path is `android/fork/libcore/.build/src-android-386/go.mod`.
`android/fork/libcore/go.mod`, `android/libneko/go.mod`, and `android/sing-box/go.mod` were inspected in CP-011 and confirmed to contain valid module declarations.
The current evidence ranks the blocker origin as:
1. generated temporary module state
2. `gomobile-matsuri` behavior
3. upstream layout assumptions
4. local dependency materialization
No upstream source files under `android/fork/` have been modified.
No repair has been attempted yet.

## TARGET_STATE
The repository has one narrowly bounded repair attempt definition that tests only the smallest evidence-backed surface for the zero-byte generated `go.mod` issue in the libcore gomobile path.
That repair surface is limited to the local `gomobile-matsuri` generation path responsible for writing the temporary `src-android-*` module roots.
When CP-012 is later executed, it must determine whether a tool-local repair is sufficient to:
- stop generating zero-byte `go.mod` files under `android/fork/libcore/.build/src-android-*`
- allow `go mod tidy -v` to succeed inside the first generated module root
- allow `android/fork/libcore/build.sh` to produce `libcore.aar`
CP-012 must explicitly stop after blocker validation.
It must not expand into broader Android build continuation.

## EXECUTION_NOTES
The smallest repair surface selected for CP-012 is the local `gomobile-matsuri` code path that emits the temporary `src-android-*` `go.mod` files.
This surface is selected because CP-011 established that:
- the upstream `libcore`, `libneko`, and `sing-box` `go.mod` files are already valid
- the failure occurs inside generated `.build/src-android-*` module roots before `build.sh` regains control
- the generated `go.mod` files exist but are zero bytes long

The first repair attempt must therefore not start by editing `android/fork/` source files.
It must test whether fixing the tool-local generated-module write path is enough to clear the blocker.

The execution sequence for CP-012 must be:
1. preserve the same session-scoped environment shape used in CP-010 and CP-011
2. create or use an isolated local copy of the `gomobile-matsuri` source used for the repair attempt
3. patch only the generated-`go.mod` write/finalization path
4. rebuild the local patched `gomobile-matsuri` and `gobind-matsuri` binaries
5. re-run only the libcore gomobile path
6. preserve `.build/` long enough to inspect the generated `src-android-*` roots
7. stop after blocker validation and document the outcome

Validation during CP-012 execution must answer all of the following:
- are the generated `src-android-*` `go.mod` files still zero bytes
- do those files now contain a valid `module` declaration
- does `go mod tidy -v` succeed inside `src-android-386`
- does `android/fork/libcore/build.sh` emit `libcore.aar`
- did the repair require any source change outside the isolated `gomobile-matsuri` repair surface

Rollback expectations if the repair attempt fails:
- restore the original `gomobile-matsuri` and `gobind-matsuri` binaries or restore the original tool selection used before the attempt
- remove the isolated patched tool workspace if it was created only for the attempt
- remove generated `.build/` contents and any failed-attempt `libcore.aar` output unless they are intentionally preserved as evidence in the report
- document the exact failure point and exact evidence in `docs/android/ANDROID_LIBCORE_GOMOBILE_REPAIR_ATTEMPT.md`
- stop without changing `android/fork/` source and without continuing into Gradle or app-level build steps

If the tool-local repair is insufficient, CP-012 must end with a documented failure and a recommendation for a new checkpoint.
It must not expand into a second repair surface inside the same checkpoint.
