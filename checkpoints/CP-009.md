## EXECUTION_NOTES
### JDK selection
AGP `8.8.1` requires JDK 17 as the minimum. JDK 21 is also compatible. The current Java 8 runtime is not sufficient. Prefer JDK 17 for maximum compatibility unless JDK 21 is already available or trivially installable.

### Android SDK installation path
Use Android SDK Command-line Tools as the entry point. A reasonable default installation path on this Windows host is `C:\Android\Sdk` or `%LOCALAPPDATA%\Android\Sdk`. Prefer whichever aligns with existing conventions on the host.

### Go version
`libcore/go.mod` declares `go 1.23.1` and `toolchain go1.23.6`. Install Go `1.23.1` or the latest `1.23.x` available. The Go toolchain directive means the runtime may auto-download `go1.23.6` on first use if the installed version is `1.23.1`.

### gomobile-matsuri / gobind-matsuri
`android/fork/libcore/init.sh` clones `AtiUsm/matsern-gomobile` and builds `gomobile-matsuri` and `gobind-matsuri` from it. This checkpoint should materialize and build these tools. The exact upstream URL and revision should be confirmed from `libcore/init.sh` before cloning.

### POSIX shell on Windows
Git for Windows ships `bin/bash.exe` under its installation directory. This is the most reliable POSIX shell on a standard Windows workstation. The `C:\Windows\system32\bash.exe` launcher (WSL) is confirmed non-functional. Document the Git Bash path and confirm that `libcore/*.sh` scripts can be invoked through it.

### External source dependency versions
The exact upstream URLs and revisions for `libneko` and `sing-box` should be determined from:
- `android/fork/libcore/go.mod` (module paths and version constraints)
- `android/fork/libcore/init.sh` or other upstream scripts that reference these repositories
- the upstream NekoBox repository's submodule or dependency metadata at the snapshot commit
If the correct revision cannot be determined with certainty, clone at the latest available state and document the revision used, noting that it may require alignment in a future checkpoint.

### Order of operations
The recommended installation sequence is:
1. JDK (unblocks Gradle and Android SDK tooling)
2. Android SDK Command-line Tools (unblocks `sdkmanager`)
3. SDK platform, Build Tools, NDK via `sdkmanager`
4. `local.properties` creation
5. Gradle wrapper cache population (allow `gradlew.bat` to download, then stop)
6. Go toolchain
7. POSIX shell confirmation (Git Bash)
8. `gomobile-matsuri` / `gobind-matsuri` build
9. `libneko` and `sing-box` materialization
10. Verification script execution

### Risk: component version drift
If the exact version of a component is unavailable (e.g., Go `1.23.1` is superseded), install the closest compatible version and document the deviation. Do not modify upstream `go.mod` or Gradle files to match.

### Risk: network-dependent steps
Most installation and materialization steps require network access. If any download fails, record the failure, skip to the next independent step, and mark the checkpoint as partial.

### Risk: Windows path length
Some Android SDK and NDK paths can exceed the Windows 260-character path limit. If encountered, document the failure and suggest a shorter SDK root.
